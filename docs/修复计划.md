## Phase 1：关键 Bug 修复（🔴 高严重度）

> 必须立即修复，影响数据正确性、运行稳定性

### 1.1 CANBusPage — publishFrame() 重复调用

- **文件**: `src/app/pages/CANBusPage.cpp`
- **问题**: TX 帧在 `onSend()` (L808) 和 `addFrameToTable()` (L994) 各调用一次 `publishFrame()`，导致 DataBus 订阅者收到重复数据
- **修复方案**: 删除 `addFrameToTable()` 末尾的 `publishFrame()` 调用，仅保留 `onSend()` 中的调用
- **验证**: 发送 TX 帧，确认 DataBus 订阅者仅收到一次

### 1.2 ProtocolDecoderPage — processData() streamOffset 错误

- **文件**: `src/app/pages/ProtocolDecoderPage.cpp`
- **问题**: `decode()` 返回的 `streamOffset` 是相对于传入 `data` 的偏移，而非相对于 `m_dataBuffer`，导致截断计算错误
- **修复方案**: 将 streamOffset 加上 `m_dataBuffer.size() - data.size()` 的偏移差，或在 decode 前记录 buffer 原始长度，截断时用 `oldSize + streamOffset` 计算
```cpp
// 修复前:
m_dataBuffer = m_dataBuffer.mid(lastEnd);

// 修复后:
int oldSize = m_dataBuffer.size() - data.size();
m_dataBuffer = m_dataBuffer.mid(oldSize + lastEnd);
```
- **验证**: 连续发送多段数据，确认帧不丢失、不重复解析

### 1.3 ProtocolDecoder — checksum remove() 偏移错位

- **文件**: `src/core/protocol/ProtocolDecoder.cpp`
- **问题**: L269-270/L430-431，`checksumInput.remove(offset, size)` 修改了 QByteArray 长度，如果 Checksum 不在末尾，后续字段偏移全部错位
- **修复方案**: 不修改原数据，改为在计算校验和时跳过 checksum 范围
```cpp
// 修复方案：分段拼接，跳过 checksum 区间
QByteArray checkData;
checkData.append(checksumInput.left(checksumIt->offset));
checkData.append(checksumInput.mid(checksumIt->offset + checksumIt->size));
quint8 calc = calcChecksum(checkData, checksumIt->algorithm, checksumIt->poly);
```
- **验证**: 测试含中间位置 checksum 字段的模板，确认校验结果正确

### 1.4 ProtocolDecoder — frameSize=0 导致 Payload 模板无法匹配

- **文件**: `src/core/protocol/ProtocolDecoder.cpp`
- **问题**: L302-341，当 `hasAutoPayload=true` 且无 Length 字段时，`frameSize=0`，L343 判断 `frameSize<=0` 直接 continue
- **修复方案**: 当 hasAutoPayload 且无 Length 字段时，设 `frameSize = -1` 作为"变长帧"标记，decode 循环中遇到 frameSize<0 时使用尾部匹配（endMarker 或剩余数据）
```cpp
if (hasAutoPayload && !hasLengthField) {
    frameSize = -1;  // 变长帧标记
}
// decode 循环中:
if (frameSize < 0) {
    // 使用 endMarker 或 headerSize + payload 推断
    frameSize = findFrameEnd(data, headerSize, searchPos);
}
```
- **验证**: 创建含 Payload 无 Length 的模板，确认能正常匹配

### 1.5 PluginLoader — unloadAll() 未注销插件

- **文件**: `src/core/plugin/PluginLoader.cpp`
- **问题**: L71-84，卸载插件库但未从 PluginRegistry 移除，导致悬空指针
- **修复方案**: 在 `unloadAll()` 中，卸载每个插件前先调用 `m_registry->unregisterPlugin(pluginName)`
```cpp
for (auto it = m_loaders.begin(); it != m_loaders.end(); ++it) {
    IPlugin* plugin = m_registry->plugin(it.key());
    if (plugin) {
        plugin->shutdown();
        m_registry->unregisterPlugin(it.key());
    }
    // ... unload library
}
```
- **验证**: 加载插件后卸载，确认 PluginRegistry 中无残留指针，访问不崩溃

### 1.6 industrial_dark.qss — 暗色主题内容错误

- **文件**: `src/resources/themes/industrial_dark.qss`
- **问题**: 内容实际是白蓝主题的副本，完全没有暗色配色
- **修复方案**: 基于 `wood_classic.qss` 重写暗色主题，关键改动：
  - 背景色: `#1E1E2E` → `#2D2D3F` 渐变
  - 文字色: `#E0E0E0` / `#CCCCCC`
  - 输入框/面板: `#2A2A3A` 背景 + `#3D3D5C` 边框
  - 按钮: `#3D5A80` 正常 / `#4A6FA5` hover
  - 选中/高亮: `#5B8BD4` / `#3D5A80`
  - 滚动条: `#4A4A5E` handle
  - 表格交替行: `#252538` / `#2A2A3A`
  - 状态色保持语义（error 红、success 绿、warning 黄）
- **验证**: 切换到 dark 主题，所有页面视觉正常、文字可读

### 1.7 SerialConsolePage — Python 脚本临时文件路径固定

- **文件**: `src/app/pages/SerialConsolePage.cpp`
- **问题**: L560，临时文件路径 `QDir::tempPath() + "/est_serial_helper.py"` 固定，多实例竞争
- **修复方案**: 改用 `QTemporaryFile` 或添加进程 ID 后缀
```cpp
// 方案 A：QTemporaryFile
QTemporaryFile tempScript(QDir::tempPath() + "/est_helper_XXXXXX.py");
tempScript.setAutoRemove(true);
if (tempScript.open()) {
    tempScript.write(helperScript.toUtf8());
    scriptPath = tempScript.fileName();
}

// 方案 B：PID 后缀（更简单）
QString scriptPath = QDir::tempPath() + QString("/est_helper_%1.py").arg(QCoreApplication::applicationPid());
```
- **验证**: 同时运行两个实例，确认不冲突
