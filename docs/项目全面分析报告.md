## 1. 整体架构概览

项目采用 **插件式单体架构（Plugin-based Modular Monolith）**，核心组件：

- **App 层**：`AppShell` 作为 QMainWindow，管理所有 Page 和导航
- **Core 层**：`DataBus`（Pub/Sub）、`TransportRegistry`（传输抽象）、`Plugin Framework`、各类 Service
- **通信机制**：DataBus 通道命名规范 `transport.<type>.<instance>` / `protocol.<type>.<channel>`

架构设计清晰，但代码中存在一些 **重复逻辑、边界条件处理不足、内存管理隐患、线程安全问题** 等需要修复。

### 2.1 应用入口 (main.cpp)

**功能**：初始化 QApplication、加载主题、创建 AppShell。

| 问题/优化点 | 严重程度 | 详细说明 |

| `logToFile` 使用 `toLocal8Bit()` 而非 `toUtf8()` | 中 | 日志文件编码依赖系统 locale，跨平台时中文字符可能乱码。应统一使用 UTF-8。 |

### 2.5 串口助手 (SerialAssistantPage)

**功能**：串口通信核心页面，含配置、收发、快捷命令、自动发送。

| 问题/优化点 | 严重程度 | 详细说明 |
|---|---|---|
| `handlePacketReceived()` 中使用 `QTimer::singleShot(0)` 转发 DataBus | 中 | 第312-314行，每次收到数据都 deferred publish。在高波特率下会产生大量 pending QTimer 事件，可能导致事件队列堆积和延迟。应使用 `Qt::QueuedConnection` 直接连接。 |
| `m_transport` 的生命周期管理不一致 | 中 | `ensureTransport()` 中创建 transport 并 `setParent(this)`，但 `~SerialAssistantPage()` 中只调用 `m_transport->close()` 不 delete。虽然 Qt 父子关系会自动 delete，但如果 transport 被提前 close，指针悬空。 |
| `sendCurrentPayload()` 中 `m_txBytes` 统计不包含失败的发送 | 低 | 第363行仅在 `m_transport->send()` 成功后累加 TX 字节，但用户可能关心"尝试发送"的统计。 |
| `loadSettings()` / `saveSettings()` 使用 `QSettings` 但没有设置路径 | 低 | 依赖 Qt 默认路径（注册表/ini），不同平台行为不一致。建议显式设置格式和路径。 |
| `refreshPorts()` 在构造时通过 `QTimer::singleShot(0)` 调用 | 低 | 第102行，延迟到事件循环。如果用户快速打开页面并点击打开，可能端口列表尚未刷新完成。 |
| `m_subscription` 的通配符订阅 `transport.serial.*` 可能收到非当前端口数据 | 低 | 回放数据通过 DataBus 发送，但如果有多个串口同时打开，所有串口数据都会进入接收区。 |
| `sendPanel` 和 `receiveView` 的 `timestampEnabled()` 初始化顺序依赖 | 低 | 第100行 `m_receiveView->setTimestampVisible(m_sendPanel->timestampEnabled())`，发送面板的状态决定了接收区的时间戳显示，这两个组件的耦合不合理。 |
| 发送失败时 `updateAutoSendState()` 未被调用 | 中 | `sendCurrentPayload()` 在发送失败后停止 autoSendTimer（第329行），但 `updateAutoSendState()` 有额外的检查逻辑。直接 stop timer 可能跳过某些状态更新。 |

### 2.10 CAN 总线 (CANBusPage)

**功能**：通过 CANable (SLCAN) 进行 CAN 帧收发。

| 问题/优化点 | 严重程度 | 详细说明 |
|---|---|---|
| `openCANable()` 中 `m_transport` 创建后没有通过 TransportRegistry | 中 | 第552-560行，先从 registry 创建，失败则直接 `new SerialTransport()`。这绕过了注册表管理，生命周期不一致。 |
| `sendSLCAN()` 发送的 `DataPacket` 没有 direction=tx | 低 | 第648-652行，metadata 中没有设置 direction，接收端无法区分是 TX 还是 RX 的 slcan 命令。 |
| `parseIncomingData()` 使用 `QString::fromLatin1()` | **高** | 第687行，串口原始字节通过 Latin-1 转为 QString。但 slcan 协议是纯 ASCII，`fromLatin1` 没问题；如果数据含非 ASCII 字节（如 CAN 数据帧的 payload），`fromLatin1` 会导致每个字节映射为 Unicode 字符，后续 `indexOf('\r')` 可能找不到正确的分隔符。应该用 `QByteArray` 直接处理。 |
| `addFrameToTable()` 中 `publishFrame()` 被调用两次 | **高** | 第993行 `publishFrame()` 在 `addFrameToTable()` 末尾又被调用一次，但第806-807行 `onSend()` 中已经调用过 `publishFrame()`。这导致 TX 帧被 publish 两次到 DataBus。 |
| `buildCANFrame()` 函数完全未使用 | 低 | 第1203-1208行定义了 `buildCANFrame()`，但代码中所有地方都直接调用 `SlcanCodec::buildDataFrame()`。 |
| `updateDlcFromPayload()` 是空壳函数 | 低 | 第1248-1251行直接调用 `updatePayloadEditorsFromDlc()`，没有额外逻辑，可删除。 |
| `m_lineBuffer` 是 `QString` 类型 | 中 | 串口数据应以 `QByteArray` 缓冲，避免编码转换问题。当前用 QString 缓冲，在二进制数据场景下会丢失信息。 |

## 4. 优化建议汇总

### 4.1 架构层面

1. **统一页面生命周期**：为所有 Page 添加 `onPageShown()` / `onPageHidden()` 虚函数，AppShell 在切换页面时调用，便于资源管理。
2. **引入内存缓存 + 异步持久化**：RecentRecordManager 应维护内存中的记录列表，通过 QTimer 延迟批量写入磁盘，减少 I/O。
3. **主题系统重构**：使用 QSS 变量（或预处理器）管理颜色，实现真正的 dark/light 切换。当前 `industrial_dark.qss` 需要完全重写。
4. **DataBus 增加调试接口**：当前只有 `subscriberCount()` 和 `activeChannels()`，建议增加消息计数、流量统计等，便于性能监控。

### 4.2 性能优化

1. **DataLogger 批量 flush**：改为每 500ms 或每 100 帧 flush 一次，或者使用 `QFile::write()` 直接写入缓冲区。
2. **SerialReceiveView 增量渲染**：`appendPacket()` 时直接 append 到 `QPlainTextEdit`，而不是同时维护 `m_entries` 向量。或者使用自定义 QAbstractListModel + QListView。
3. **CANBusPage 帧表格使用模型/视图**：将 `QTableWidget` 替换为 `QTableView` + 自定义 `QAbstractTableModel`，避免每次添加帧都创建大量 `QTableWidgetItem`。
4. **ProtocolDecoderPage 增量更新**：`updateFrameList()` 只追加新增帧，不清空重建。

### 4.3 可靠性优化

1. **PluginLoader 卸载前注销插件**：在 `unloadAll()` 中遍历 `m_loaders`，找到对应的插件名并从 `PluginRegistry` 注销。
2. **SerialTransport 帧缓冲**：在 SerialTransport 或上层增加面向帧的缓冲逻辑，确保 `dataReceived` 信号每次 emit 完整帧。
3. **多实例文件竞争**：SerialConsolePage 的 Python 脚本、DataLogger 的日志文件、RecentRecordManager 的 JSON 文件都应使用带进程 ID 或临时文件名的路径。
4. **编码选择器**：SerialConsolePage 和 SerialReceiveView 应提供编码选择（UTF-8/GBK/ASCII/Hex），避免硬编码解码方式。

### 4.4 代码质量优化

1. **统一 QSettings 路径**：所有模块使用相同的 organizationName 和 applicationName。
2. **删除未使用的代码**：`buildCANFrame()`、`updateDlcFromPayload()`、`m_byteCountLabel`、`m_statusIndicator` 等。
3. **日志系统**：引入分级日志（debug/info/warning/error），发布版本禁用 debug 输出。
4. **单元测试**：核心模块（DataBus、ProtocolDecoder、SlcanCodec、RecentRecordManager）应补充单元测试。
